## Summary of Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Config location | **Builder repo first**, content repo override | Leon controls both repos now; content repo override enables future multi-user |
| Theme structure | **Full template sets per theme**, shared base partials | Buffbaby and marbles have different headers, footers, page layouts — CSS-only theming isn't enough |
| `src/` directory | **Entirely generated at build time** (gitignored) | Clean separation: theme files checked into `themes/`, content generated by preprocessor, `src/` assembled at build time |
| Eleventy config | **Single `eleventy.config.js`** reads site config dynamically | One entry point, config-driven. Per-theme sub-configs can be added later |
| Preprocessing plugins | **Registry pattern** — config lists which preprocessors to run, builder loads them | New preprocessors can be added as files in `scripts/utils/` and activated per-site via config |
| Theme storage | **Filesystem in builder repo** for now, database-backed later | Keeps it simple; the config schema supports a future `source: "registry"` field for shared/forked themes |

### Build Flow (Updated)

```
1. Load site config
   ├── Check content repo for bloob.config.yaml
   └── Fall back to sites/{site-name}.yaml in builder repo

2. Assemble src/ (NEW STEP)
   ├── Clean src/ directory
   ├── Copy themes/_base/ → src/_includes/ and src/assets/
   ├── Copy themes/{theme}/ → src/_includes/ and src/assets/ (overrides _base)
   ├── Copy themes/{theme}/pages/ → src/ (homepage, tag pages, etc.)
   └── Generate src/_data/site.js from config

3. Clone content repo (existing)

4. Preprocess content (updated)
   ├── Run core preprocessors (always)
   └── Run plugin preprocessors (from config list)

5. Generate OG images (existing)

6. Build Eleventy (existing, but config-driven)
   ├── Collections registered based on site config
   └── Visualizers activated based on site config

7. Run Pagefind (existing)

8. Output: _site/ ready for deployment
```

## Site Config Schema

### `sites/buffbaby.yaml` (in builder repo)

```yaml
# Bloob Haus Site Configuration
# This file drives the entire build process for a single site.

site:
  name: "Buff Baby Kitchen"
  description: "Home-cooked recipes from our kitchen"
  url: "https://buffbaby.bloob.haus"
  subdomain: buffbaby
  author: "Leon"
  language: "en-us"

content:
  repo: "leonsanten/buffbaby"        # GitHub repo (owner/name)
  branch: "main"                      # branch to clone
  publish_mode: "blocklist"           # "blocklist" or "allowlist"
  blocklist_tag: "not-for-public"     # tag used for blocklist mode

theme: "warm-kitchen"                 # theme folder name in themes/

# Preprocessing plugins to run IN ADDITION to core pipeline
# Core always runs: publish-filter, file-index-builder, wiki-link-resolver,
#   markdown-link-resolver, attachment-resolver, transclusion-handler,
#   comment-stripper, git-date-extractor, tag-extractor
preprocessing:
  plugins: []
  # Future example: plugins: ["cooklang-parser", "recipe-scaler"]

# Which visualizers to activate for this site
visualizers:
  - checkbox-tracker
  # Future: - recipe-scaler
  # Future: - bubble-map

# Eleventy collection configuration
collections:
  # Auto-detect sections from top-level folders (current behavior)
  auto_sections: true
  # Additional named collections (optional)
  # custom:
  #   - name: "recipesByDate"
  #     source: "recipes"
  #     sort: "date"
  #     reverse: true

# Media/image optimization settings
media:
  optimize: true
  formats: ["webp", "jpeg"]
  widths: [600, 1200]

# Features to enable
features:
  rss: true
  sitemap: true
  robots_txt: true
  search: true          # Pagefind
  backlinks: true
  og_images: true
  tags: true
  custom_404: true

# Permalink style
permalinks:
  slugify: true         # lowercase + hyphens
```

### `bloob.config.yaml` (in content repo — optional override)

```yaml
# Optional: content repo can override builder-repo config
# Only specified fields override; everything else uses builder config

site:
  name: "Buff Baby Kitchen"
  description: "Updated description from content repo"

# Could also override theme, features, etc.
# theme: "custom-fork"
```

### `themes/warm-kitchen/theme.yaml` (theme metadata)

```yaml
# Theme metadata
name: "Warm Kitchen"
description: "A warm, inviting theme for recipe and food blogs"
author: "Leon"
version: "1.0.0"

# Default features this theme supports well
# (site config can override)
supports:
  sections: true        # auto-generated section navigation
  card_grid: true       # card-style content listings
  backlinks: true
  tags: true
  search: true

# Theme-specific defaults (site config overrides these)
defaults:
  collections:
    auto_sections: true
```

### Step T5: Update `preprocess-content.js` for Plugin Architecture (1–2 hours)

**Goal:** Preprocessing pipeline runs core steps always, plus optional plugins from config

**Tasks:**

- [ ] **T5.1** Define the preprocessor plugin interface:
  ```js

    // filePath: absolute path to the .md file
    // content: file content string (after frontmatter extraction)
    // frontmatter: parsed frontmatter object
    // context: { fileIndex, attachmentIndex, config, stats }
    //
    // Returns: { content, frontmatter } (modified)
  }
  ```

- [ ] **T5.2** Refactor existing preprocessing to separate core from plugins:
  ```js

    'file-index-builder',
    'comment-stripper',

    'wiki-link-resolver',
    'markdown-link-resolver',
    'attachment-resolver',
  ```

  ```js
      const plugin = await import(pluginPath);
      await plugin.process(filePath, content, frontmatter, context);

      console.warn(`⚠️  Plugin not found: ${pluginName}`);
    }
  }
  ```

## Future: Theme Storage in Database

The current architecture stores themes as files in `themes/`. For future multi-user support where users can fork and customize themes, the path forward is:

1. **Theme Registry API** — themes stored in a database (or Git repos) with metadata
2. **Site config gains `theme_source` field:**
   ```yaml
   theme: "warm-kitchen"
   theme_source: "builtin"    # or "registry" or "custom"
   theme_repo: ""             # Git repo URL for custom themes
   ```
3. **At build time**, the builder fetches the theme from the appropriate source before assembling `src/`
4. **Users can fork a builtin theme** — creates a copy in their own storage that they can edit

This requires no changes to the theme *structure* — just changes to where themes are fetched from. The filesystem layout in `themes/` would remain the contract.
